<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Server Actions | Login credentials, password reset requests cross from client |
| Email link -> Auth callback | Password reset token in URL, could be intercepted |
| Protected page -> Supabase | User data fetch trusts Supabase session validity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | signIn action | mitigate | Server-side validation via Supabase Auth (bcrypt password check); rate limited by Supabase |
| T-02-02 | Information Disclosure | login error messages | mitigate | Generic "Wrong email or password" message; do not reveal which field is wrong |
| T-02-03 | Spoofing | password reset | mitigate | Supabase handles token generation and validation; redirectTo is validated server-side |
| T-02-04 | Tampering | email_confirmed_at display | accept | UI badge is informational only; actual enforcement is in RLS policies (Plan 01) |
| T-02-05 | Repudiation | signOut action | accept | Logout is a user-initiated action; no audit trail needed for V1 |
</threat_model>
