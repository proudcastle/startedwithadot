<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Server Actions | Untrusted form data crosses from client to server |
| Server -> Supabase Auth | Auth mutations (signUp) cross to external service |
| Supabase Auth -> PostgreSQL | Trigger fires in auth schema, writes to public schema |
| Browser -> Supabase (direct) | Username availability check bypasses server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | signUp server action | mitigate | Validate all inputs with Zod server-side before calling supabase.auth.signUp; do not trust client validation alone |
| T-01-02 | Tampering | username field | mitigate | DB constraint (CHECK + UNIQUE INDEX on LOWER) is source of truth; client validation is UX only |
| T-01-03 | Information Disclosure | username availability check | mitigate | Return only "taken" or "available" -- do not reveal email or other profile data in availability response |
| T-01-04 | Elevation of Privilege | is_admin flag | mitigate | is_admin DEFAULT FALSE in DB, no client-side mechanism to set it; only modifiable via Supabase Dashboard SQL |
| T-01-05 | Spoofing | auth state in header | mitigate | Use getUser() (server-validated) not getSession() (client JWT, spoofable) per RESEARCH.md |
| T-01-06 | Tampering | profile trigger | mitigate | handle_new_user runs as SECURITY DEFINER; username format/length validated by DB CHECK constraints |
| T-01-07 | Denial of Service | signup endpoint | accept | Supabase Auth has built-in rate limiting on signup; no custom rate limit needed in Phase 1 |
</threat_model>
