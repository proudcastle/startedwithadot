<objective>
Add admin curation capabilities: status changes, proposal deletion, and version creation linked to implemented proposals. This completes the propose-vote-curate cycle -- the admin can now accept the best proposals and mark them as implemented versions.

Purpose: The admin closing the loop is what makes the community feedback meaningful. Without curation, proposals are just a wish list. With it, the dot evolves.
Output: Admin dropdown on proposal cards with Accept/Reject/Implement/Delete actions, version creation dialog, and the supporting database migration.
</objective>

## Phase Goal

**As a** community member, **I want to** propose ideas for the dot, vote on others' proposals, and see an admin curate the best ones, **so that** the community decides what the dot becomes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client admin-menu -> Server Actions | Admin status changes and deletions cross from client to server |
| Server Actions -> Database | RLS policies are the authoritative enforcement layer for admin checks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Elevation of Privilege | admin-menu.tsx | mitigate | is_admin read from profile for UI rendering, but RLS policies are the real enforcement; even if client is tampered, DB rejects non-admin mutations |
| T-02-09 | Tampering | deleteProposal | mitigate | RLS DELETE policy checks is_admin via profiles table; window.confirm adds UI-level friction |
| T-02-10 | Tampering | updateProposalStatus | mitigate | RLS UPDATE policy checks is_admin via profiles table; status value constrained to enum type |
| T-02-11 | Tampering | createVersion | mitigate | RLS INSERT on versions checks is_admin; Zod validates version_number and title fields |
</threat_model>
