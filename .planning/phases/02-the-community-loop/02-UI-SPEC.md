## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline dot symbol spacing, badge internal padding |
| sm | 8px | Compact element spacing, input internal padding, gap between dot symbols in vote counter |
| md | 16px | Default element spacing, card internal padding, gap between proposal cards |
| lg | 24px | Section padding, space between status tabs and first card, space between input area and feed |
| xl | 32px | Layout gaps, page horizontal padding on mobile |
| 2xl | 48px | Major section breaks, page vertical padding |
| 3xl | 64px | Page-level top spacing (below header) |

Exceptions: 44px minimum touch target for the vote button (accessibility requirement, not in spacing scale but enforced as `min-w-[44px] min-h-[44px]`).

Source: default 8-point scale; exception for WCAG 2.5.5 touch target

## Typography

| Role | Font | Size | Weight | Line Height | Usage |
|------|------|------|--------|-------------|-------|
| Body | Geist | 16px (text-base) | 400 (normal) | 1.5 | Proposal text, descriptions, form labels |
| Small | Geist | 14px (text-sm) | 400 (normal) | 1.5 | Timestamps, character counter, vote counts numeric, helper text |
| Heading | Press Start 2P | 20px (text-xl) | 400 (only weight available) | 1.2 | Page title ("Proposals"), section headings |
| Display | Press Start 2P | 12px (text-xs) | 400 (only weight available) | 1.5 | Status tab labels, status badge text, empty state heading |

Font weight constraint: Geist uses 400 only. Press Start 2P has only 400 available. No semibold or bold in this phase -- visual emphasis comes from color contrast (foreground vs muted-foreground) and font family switching (Geist vs Press Start 2P), not weight.

Note on Press Start 2P sizing: This font renders approximately 2x larger than Geist at the same pixel size. 12px Press Start 2P reads visually equivalent to ~20px Geist. 20px Press Start 2P reads as a large heading. Never use Press Start 2P below 10px -- it becomes illegible.

Source: CLAUDE.md font constraints, app/layout.tsx font loading, REQUIREMENTS.md PROP-02

## Color

All colors are monochrome (project constraint: "Strictly monochrome black/white/grays until the community introduces color through proposals").

| Role | CSS Variable | HSL Value | Usage |
|------|-------------|-----------|-------|
| Dominant (60%) | `--background` | 0 0% 0% (pure black) | Page background, canvas background |
| Secondary (30%) | `--card` | 0 0% 4% (near-black) | Proposal cards, popover/dropdown surfaces, dialog |
| Accent (10%) | `--foreground` | 0 0% 100% (pure white) | See reserved list below |
| Muted | `--muted-foreground` | 0 0% 64% (mid-gray) | Timestamps, unvoted dots, placeholder text, secondary info |
| Border | `--border` | 0 0% 15% (dark gray) | Card borders, input borders, separator lines |
| Destructive | `--destructive` | 0 62% 30% (dark red) | Delete proposal action only |

Accent (white foreground) reserved for:
1. Proposal text (primary content)
2. Author username
3. Active vote dot (voted state)
4. Active status tab indicator
5. Submit button text (on primary background)
6. Admin dropdown trigger icon
7. Status badge text for "Implemented" status
8. Page heading text

Everything else uses `--muted-foreground` (gray) or `--border` (dark gray).

Source: app/globals.css CSS variables (Phase 1 codebase), CLAUDE.md monochrome constraint

## Interaction Contracts

### Proposal Submission (PROP-01, PROP-02, PROP-06)

| State | Visual | Behavior |
|-------|--------|----------|
| Default (auth, verified) | Input field visible with placeholder: "What should the dot do next?" | Character counter shows "140" in muted-foreground |
| Typing | Character counter counts down, color stays muted-foreground | At 20 remaining: counter text changes to foreground (white) |
| At limit (0 remaining) | Counter shows "0" in foreground, input prevents further typing | maxLength enforced on input element |
| Submitting | Submit button shows pending state (existing SubmitButton pattern) | Input disabled during submission |
| Success | Input clears, toast: "Proposal submitted. The dot is listening." | Feed revalidates, new proposal appears |
| Rate limited | Toast: "Three ideas a day. That's the limit. Quality over quantity." | Input remains, text preserved |
| Error (server) | Toast: "Something broke. The dot is investigating." | Input remains, text preserved |
| Unauthenticated | Input area replaced with CTA text + sign-up link | "Got an idea? Sign up to tell the dot." with link to /sign-up |
| Unverified email | Input area replaced with verification prompt | "Verify your email first. The dot needs to know you're real." |

Character counter narrator milestones (source: CONTEXT.md D-04, Specifics section):
- 140 remaining: "140 characters. Make them count."
- 50 remaining: "Getting tight..."
- 20 remaining: "Almost out. Choose wisely."
- 0 remaining: "That's it. Submit or trim."

### Vote Toggle (VOTE-01, VOTE-02, VOTE-03, VOTE-04)

| State | Visual | Behavior |
|-------|--------|----------|
| Not voted (auth) | Muted dot(s) in `muted-foreground` color | Cursor pointer, 44px min touch target |
| Voted (own) | Dot(s) in `foreground` (white) color | Click to unvote |
| Optimistic update | Count adjusts +/-1 immediately | No loading spinner; if server fails, rollback silently |
| Count <= 10 | Individual dot symbols: "&#9679; &#9679; &#9679;" | Dots spaced 8px apart |
| Count > 10 | Single dot + numeric: "&#9679; 42" | Numeric in 14px Geist |
| Count = 0 | Single hollow circle: "&#9675;" | Indicates no votes yet |
| Unauthenticated | Dots visible but not clickable | Cursor default, no hover effect |
| Pending (during server call) | Button disabled, no visual change (optimistic already applied) | Prevents double-click |

Source: CONTEXT.md D-06, D-07, D-08; REQUIREMENTS.md VOTE-01 through VOTE-04

### Status Tabs (PROP-05)

| Tab | Filter | Default |
|-----|--------|---------|
| All | No filter applied | Yes (default active) |
| Open | `status = 'open'` | No |
| Accepted | `status = 'accepted'` | No |
| Implemented | `status = 'implemented'` | No |

Tab state stored in URL search params (`?status=open`) for shareability. "Rejected" proposals are not shown in any tab -- they are hidden from the public feed.

Active tab: `data-[state=active]:bg-accent data-[state=active]:text-foreground` (built into 8bitcn Tabs component).
Inactive tab: `text-muted-foreground`.

Source: CONTEXT.md D-01, D-02; RESEARCH.md open question 2

### Admin Dropdown (ADMN-01, ADMN-03, ADMN-04)

| Action | Label | Behavior |
|--------|-------|----------|
| Accept | "Accept" | Status changes to accepted, feed revalidates, toast: "Proposal accepted." |
| Reject | "Reject" | Status changes to rejected, card disappears from feed, toast: "Proposal rejected." |
| Mark Implemented | "Mark Implemented" | Opens Dialog for version creation (version number + title fields), then status changes to implemented |
| Delete | "Delete" | Confirmation step required (see Destructive Confirmation below), then proposal removed |

Trigger: three-dot icon button (&#x22EF;) using Lucide `MoreHorizontal` icon. Visible only when `is_admin = true` on user profile. Positioned at top-right of proposal card.

Source: CONTEXT.md D-09, D-10, D-11

### Version Creation Dialog (ADMN-02)

| Field | Type | Validation |
|-------|------|------------|
| Version number | Text input | Required, max 20 chars (e.g., "v0.2") |
| Title | Text input | Required, max 100 chars (e.g., "The Dot Moves") |

Dialog uses 8bitcn Dialog component. Submit creates version linked to proposal, changes proposal status to "implemented", closes dialog, revalidates feed.

Source: CONTEXT.md D-10; RESEARCH.md versionSchema

## Destructive Actions

| Action | Trigger | Confirmation Method | Confirmation Copy |
|--------|---------|--------------------|--------------------|
| Delete proposal (admin) | Admin dropdown "Delete" | Browser `window.confirm()` dialog | "Delete this proposal? This can't be undone." |

Rationale for `window.confirm()` over custom modal: This is an admin-only action with low frequency. A native confirm dialog is sufficient for V1 MVP. Phase 4 can upgrade to a styled confirmation if needed.

### 8bitcn Vetting Evidence

**Tabs component** (`@8bitcn/tabs`): Viewed via `npx shadcn@latest view @8bitcn/tabs`. Pure UI wrapper around shadcn Tabs (Radix). Uses `class-variance-authority` for variants and `cn()` for class merging. References `retro.css` for font styling. No `fetch`, `eval`, `process.env`, or dynamic imports found. Clean.

**Dialog component** (`@8bitcn/dialog`): Viewed via `npx shadcn@latest view @8bitcn/dialog`. Pure UI wrapper around shadcn Dialog (Radix). Uses `class-variance-authority` for variants and `cn()` for class merging. References `retro.css` for font styling. No `fetch`, `eval`, `process.env`, or dynamic imports found. Clean.

**Note on retro.css:** The 8bitcn `retro.css` file includes `@import url("https://fonts.googleapis.com/css2?family=Press+Start+2P&display=swap")`. This is a Google Fonts import, which is redundant since the project already loads Press Start 2P via `next/font/google` in `app/layout.tsx`. This import should be removed after installation to avoid a double-load and CLS penalty. The font is already available via `--font-press-start-2p` CSS variable.

Source: Direct `shadcn view` output, 2026-05-18

## Proposal Card Layout Spec

```
+--------------------------------------------------+
| [Status Badge: "Open"]            [...] (admin)  |
|                                                   |
| Proposal text goes here, up to 140 characters     |
| maximum. This is the main content of the card.    |
|                                                   |
| @username  ·  2h ago      [Vote: ●●●  or  ● 42] |
+--------------------------------------------------+
```

- Card uses 8bitcn Card with 16px internal padding
- Status badge top-left, admin menu trigger top-right (only if admin)
- Proposal text in 16px Geist, foreground color
- Bottom row: username (14px, muted-foreground), dot separator, relative timestamp (14px, muted-foreground), vote button flush right
- Cards stack vertically with 16px gap between them
- Card background: `--card` (0 0% 4%)
- Card border: `--border` (0 0% 15%), rendered via 8bitcn pixel-art border styling
