<objective>
Deliver the core proposal submission and feed display as a complete vertical slice. After this plan, an authenticated user can submit a 140-character proposal and see it appear in a sorted feed alongside other proposals. Unauthenticated visitors can browse the feed but not submit.

Output: Working /proposals page with inline submission form and card-based feed.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Server Action | Proposal text from untrusted user input crosses to server |
| Server Action -> Database | Validated data written to Supabase with RLS enforcement |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | actions/proposals.ts | mitigate | Zod validation enforces 140-char max and non-empty; RLS enforces auth.uid() = user_id and email_confirmed_at check |
| T-02-02 | Tampering | Rate limit bypass | mitigate | Server-side count query in submitProposal checks proposals created today (UTC); RLS prevents unauthenticated inserts as backstop |
| T-02-03 | Spoofing | Proposal submission | mitigate | supabase.auth.getUser() verifies JWT server-side; user_id set from auth, not form data |
| T-02-04 | Info Disclosure | Proposal feed | accept | All proposals are public by design (RLS SELECT allows all); no PII beyond username |
</threat_model>
